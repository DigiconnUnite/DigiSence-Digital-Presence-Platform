Critical bugs found
CRITICAL Auth context race condition — src/app/login/business/page.tsx
After login(), the page immediately calls fetch('/api/business') without waiting for the auth cookie to propagate. This causes intermittent 401s on first login — the user sees a "Failed to load business dashboard" error even though credentials are correct.
// Bug: immediate fetch after login, cookie may not yet be set const result = await login(email, password) if (result.success) { const response = await fetch('/api/business') // ← races the cookie // Fix: pass credentials explicitly or wait for auth refresh await refreshAuth() const response = await fetch('/api/business', { credentials: 'include' })
CRITICAL Password reset route resets the WRONG user — src/app/api/admin/password-reset/route.ts
The admin password-reset endpoint uses admin.userId (the calling super admin) to look up a business, then resets that business admin's password. But the super admin might not have a business. The query returns null and throws a 404 instead of resetting the target user. The route is missing a businessId or targetUserId parameter entirely.
// Bug: looks up business for the CALLER (super admin), not the target const business = await db.business.findFirst({ where: { adminId: admin.userId } }) // Fix: accept targetUserId in request body const { targetUserId, newPassword } = body await db.user.update({ where: { id: targetUserId }, data: { password: hashedPassword } })
CRITICAL global.io Socket.IO not typed — crashes in TS strict mode
Every API route checks if (global.io) but global.io is never declared in the global type namespace. This compiles only because the project isn't running strict TypeScript. In production builds with noImplicitAny enabled this will fail at compile time.
// Missing in types/global.d.ts (file doesn't exist) declare global { var io: import('socket.io').Server | undefined }
CRITICAL Business delete doesn't delete the user — api/admin/businesses/route.ts DELETE handler
The root-level DELETE (from query param ?id=...) deletes products and inquiries but does NOT delete the associated admin user. This leaves orphaned user accounts. The api/admin/businesses/[id]/route.ts DELETE does it correctly — inconsistency between the two delete paths.

forceRefresh() queries /api/businesses?slug=X but the businesses API doesn't support a slug param — it only supports search, page, etc. The fetch always returns the full list, not the current business, so the business state gets silently replaced with the first page of ALL businesses.
HIGH useInfiniteQuery imported after use — src/lib/hooks/useAuditLog.ts
useAuditLogsInfinite calls useInfiniteQuery before the import statement at the bottom of the file. This is a runtime error in some bundler configurations. Move the import to the top of the file.
HIGH AuditLog pagination buttons are always disabled
AuditLogTable.tsx renders Previous/Next buttons with disabled hardcoded — they never enable. There is no pagination logic. With 50+ audit log entries, users cannot navigate past the first page.
HIGH Contact form has no submission handler

API latency & performance issues
CRITICAL N+1 query in slug uniqueness check — generateUniqueSlug()
generateUniqueSlug() is defined identically in 3 files and does sequential DB reads in a while loop. For a business named "Tech Corp" it queries: findFirst({slug:'tech-corp'}), then findFirst({slug:'tech-corp-1'}), etc. Under high import load this hammers the DB.
// Fix: one query to find all conflicting slugs at once const existing = await db.business.findMany({ where: { slug: { startsWith: baseSlug } }, select: { slug: true } }) const slugSet = new Set(existing.map(b => b.slug)) let slug = baseSlug, i = 1 while (slugSet.has(slug)) slug = `${baseSlug}-${i++}`
CRITICAL hashPassword() dynamically imported inside DB transactions
In multiple routes: const { hashPassword } = await import('@/lib/auth') inside a db.$transaction(). Dynamic imports can take 50–200ms in cold-start scenarios. Meanwhile the DB transaction is open and holding a connection. This increases connection pool pressure and risks transaction timeouts under load.
// Fix: import at module top-level import { hashPassword } from '@/lib/auth' // Then inside the handler: const hashedPassword = await hashPassword(password) const result = await db.$transaction(async (tx) => { ... })
CRITICAL Bulk delete uses a for-loop inside a transaction
api/admin/businesses/bulk/delete/route.ts iterates each business individually: for (const business of businessesToDelete) { await tx.product.deleteMany(...); await tx.inquiry.deleteMany(...); await tx.business.delete(...) }. For 50 businesses this is 150 sequential DB calls inside one transaction. Use deleteMany with an in filter instead.
// Fix: batch deletes await tx.product.deleteMany({ where: { businessId: { in: businessIds } } }) await tx.inquiry.deleteMany({ where: { businessId: { in: businessIds } } }) await tx.business.deleteMany({ where: { id: { in: businessIds } } }) await tx.user.deleteMany({ where: { id: { in: adminIds } } })
HIGH Admin dashboard fetches admin-stats and businesses separately — 2 round trips
On page load the admin dashboard fires both /api/admin/businesses and an inferred stats query. MongoDB aggregations are expensive on large collections. These should be combined into a single aggregation pipeline endpoint.
HIGH Professional login fetches ALL professionals and filters client-side
login/professional/page.tsx: fetch('/api/professionals') returns every professional, then does data.professionals.find(p => p.adminId === result.user?.id) client-side. As professional count grows this becomes an unbounded payload. Add a ?adminId= query param to the professionals API.
// Fix: server-side filter fetch(`/api/professionals?adminId=${result.user.id}`) // In API: where: { adminId: userId, isActive: true }
HIGH Business catalog page: double DB call for metadata
src/app/catalog/[business]/page.tsx calls db.business.findUnique() twice — once in the page function and once in generateMetadata(). Next.js can cache this via cache() or React's built-in request deduplication if the same function is used.
import { cache } from 'react' const getBusiness = cache((slug: string) => db.business.findUnique({ where: { slug } })) // Use getBusiness() in both page() and generateMetadata()
HIGH No rate limiting on inquiry submission or OTP endpoints
/api/inquiries, /api/auth/forgot-password, and /api/auth/verify-otp have no rate limiting. An attacker can spam inquiries or brute-force OTPs. Add middleware using next-rate-limit or Upstash Redis.
MEDIUM console.log() calls in production API routes
Over 40 console.log() calls found in API routes including: business category debug logs, products count logging, and "Admin API returning businesses: X of Y" on every admin list request. These serialize objects on every request — remove or gate behind process.env.NODE_ENV === 'development'.
MEDIUM Business listing page fetches & filters client-side
src/app/businesses/page.tsx fetches all businesses at once (/api/businesses with no pagination params), then filters and searches client-side. As the platform grows, this will send megabytes of JSON on each page load. Move search and category filtering to the server using query params.
MEDIUM Aurora component runs requestAnimationFrame loop on all pages
src/components/Aurora.tsx runs a continuous requestAnimationFrame loop updating hue state 60 times per second. This is a background CPU drain on every page that imports it. The visual output is a simple CSS gradient — replace with a CSS animation.
Frontend issues
CRITICAL Hydration mismatch — BusinessProfile.tsx
isLoading is initialized to true on the server but set to false in useEffect. The server renders the skeleton while the client renders the content — React hydration will throw a mismatch warning and potentially flash the skeleton on first paint. Initialize to false since data is already available via SSR props.
HIGH Missing Suspense boundary on useSearchParams() — businesses & professionals pages
Both listing pages correctly wrap content in <Suspense> — good. But the inner component reads useSearchParams() which requires a Suspense boundary in Next.js 13+. The fallback skeleton in BusinessesLoading doesn't maintain layout shift prevention (different height than loaded content).
HIGH Touch state persists across mouse events — hero slider bug
The hero slider in BusinessProfile.tsx uses the same touchStart/touchEnd state for both touch and mouse events. If a user does a slow mouse drag (not a swipe), the stale state from a previous touch can accidentally trigger a slide change. Clear state in the correct event handlers.
HIGH Theme saved to localStorage but not synced on mount — ThemeContext
ThemeContext loads theme from localStorage in a useEffect. This means every professional dashboard renders with default styles first, then re-renders with saved styles. Causes a flash of unstyled content. Use Next.js next/headers cookies or render a blocking script to set CSS variables before React hydrates.
MEDIUM Alert dialog uses native alert() calls in 12+ places
BusinessProfile.tsx uses alert() for inquiry success/error messages, product share errors, and phone/email availability checks. Native browser alerts block the main thread and look inconsistent. Replace with the existing useToast() hook or the UnifiedModal component.
MEDIUM Missing key prop stability — product carousel
The Carousel in product section uses key={product.id} — correct. But the grid view is rendered inside the same IIFE as ProductCard, so if view mode toggles, all cards unmount. Extract the view toggle outside the IIFE and memoize ProductCard.
MEDIUM img tags used instead of Next.js Image throughout
The majority of business logos, product images, and banner images use bare <img> tags instead of Next.js <Image>. This means no automatic WebP conversion, no layout shift prevention, no built-in lazy loading optimization. Especially impactful on the businesses listing page with 20+ logos.

The exact same getSuperAdmin() function (13 lines) is copy-pasted into at least 15 route files. Same for generateUniqueSlug() in 3 files. Any bug in the auth check logic must be fixed in 15 places. Extract to src/lib/auth-helpers.ts.
// src/lib/auth-helpers.ts export async function requireSuperAdmin(request: NextRequest) { const token = getTokenFromRequest(request) ?? request.cookies.get('auth-token')?.value if (!token) return null const payload = verifyToken(token) return payload?.role === 'SUPER_ADMIN' ? payload : null }

In registration-inquiries/route.ts: fetch(`${process.env.NEXT_PUBLIC_API_URL}/api/notifications`, ...). An API route calling another API route via HTTP is an anti-pattern — it adds network overhead, fails in local dev without the env var, and can cause circular timeouts. Call the notification logic directly as a function.
CRITICAL Upload route has no authentication
src/app/api/upload/route.ts accepts any file upload from anyone — no auth check, no user ID binding. Any anonymous user can upload files to your S3 bucket. Add auth verification before processing uploads.

Consistency issues (frontend ↔ backend)
CRITICAL API response shape inconsistency — pagination field names differ
Admin businesses API returns { pagination: { total, totalPages } }. The TypeScript interfaces in useBusinesses.ts and useProfessionals.ts expect { pagination: { totalItems, totalPages } }. This mismatch means pagination display is always incorrect for professionals — showing 0 total items.
// API returns: { total: 42 } // Frontend expects: { totalItems: 42 } // Pick one and standardize across ALL paginated endpoints
CRITICAL Two duplicate bulkStatusSchema and bulkDeleteSchema in the same route file
api/admin/businesses/route.ts defines both schemas AND handles bulk operations in the root PATCH handler. There are ALSO separate route files at bulk/status/route.ts and bulk/delete/route.ts. Two implementations exist simultaneously. The frontend might hit either depending on the path. Remove the root-level bulk handlers.
HIGH Professional update uses wrong API path
useProfessionals.ts sends updates to /api/professionals/{id} (public endpoint) but deletes to the same path. The admin toggle uses /api/professionals/{id} via PUT in the admin route — but the public professionals API and admin professionals API are separate routes. Updates and deletes should go through /api/admin/professionals/{id}.
HIGH Error response format is inconsistent
Some routes return { error: 'message' }, others return { message: 'error' }, and some return { error: 'message', details: [...] }. Frontend error handling catches different keys in different places. Standardize to { error: string, details?: any } across all routes.
HIGH Category type filtering inconsistency
Admin categories API filters by type: 'BUSINESS' always. Public categories API returns all types. Business dashboard category dropdown includes PRODUCT categories alongside BUSINESS categories. The distinction between category types is not consistently enforced across the UI.
MEDIUM Socket.IO events use different naming conventions
Events emitted: business-created, business-updated, business-bulk-deleted, businesses-bulk-status-updated. Note the plural/singular inconsistency (business- vs businesses-). The client socket hook listens for specific event names — any mismatch means real-time updates silently fail.
MEDIUM Design token inconsistency — two separate color systems
The dashboard and admin pages use Tailwind slate-*/orange-* classes directly. The UI components use CSS variables (var(--color-text-primary)). Light/dark mode only works for components using CSS variables. A unified design token layer is missing.
MEDIUM Registration inquiry approval endpoint doesn't exist
useRegistrationInquiries.ts calls fetch('/api/admin/registration-inquiries/{id}/approve', { method: 'POST' }) and .../reject. Neither of these sub-routes exist in the codebase. The approve/reject buttons in the admin panel silently return 404s.
